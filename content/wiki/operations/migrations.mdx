## Running migrations

In Vercel-hosted prod, we apply via the Supabase MCP `apply_migration` tool. The migration file is committed to git so the schema is reproducible; the actual DB change is run separately.

Local dev: `supabase db push` against your local CLI-managed instance (or copy the SQL into the dashboard).

## Convention

- One migration per concern. If a feature needs 3 columns + 1 index, that's ONE file.
- Idempotent where reasonable (`if not exists`, `on conflict do nothing`). Migration 0016 uses `add column if not exists`, conditional CHECK creation via a DO block.
- Number monotonically. Never rewrite a merged migration; add a new one to fix it.
